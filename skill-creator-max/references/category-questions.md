# カテゴリ別質問集

新しいスキルを設計する際に、カテゴリに応じて検討すべき質問をまとめたチェックリスト。

## テスト・検証系

このプロジェクトの例: `/run-tests`, `/css-lint`, `/ui-test`, `/perf-test`, `/i18n-check`

### 対象範囲

- [ ] 何をテストするか？（ユニット / 統合 / UI / パフォーマンス / リント）
- [ ] テスト対象のファイル・ディレクトリは？
- [ ] テストフレームワーク・ツールは何か？（Vitest, Chrome MCP, bash script）

### 実行条件

- [ ] テスト前に必要な前提条件は？（ビルド、サーバー起動など）
- [ ] 環境依存はあるか？（Node.js バージョン、ブラウザなど）
- [ ] ローカル vs 本番で挙動が変わるか？

### 合否基準

- [ ] 合格/不合格の明確な基準はあるか？
- [ ] 数値目標はあるか？（例: FCP < 1000ms, CLS < 0.1）
- [ ] 警告レベル（⚠️）とエラーレベル（❌）の区別はあるか？

### 失敗時

- [ ] よくある失敗パターンは列挙できるか？
- [ ] 各失敗パターンの対処法は明確か？
- [ ] 自動修正は可能か？（修正提案 vs 手動修正）

### レポート

- [ ] 結果のレポート形式は定義されているか？
- [ ] 成功/失敗の絵文字表記（✅/⚠️/❌）は統一されているか？

## ビルド・デプロイ系

このプロジェクトの例: `/deploy`, `/deploy-test`

### フロー

- [ ] デプロイの前提条件チェック（型チェック、リント）はあるか？
- [ ] CI パイプラインとの連携方法は？
- [ ] デプロイ先の環境は？（Vercel, AWS, etc.）

### 安全性

- [ ] dry-run / preview モードはあるか？
- [ ] ロールバック手順は定義されているか？
- [ ] 本番影響のある操作に確認ステップはあるか？

### 待機・リトライ

- [ ] CI 完了までの待機方法は？
- [ ] タイムアウト時間は設定されているか？
- [ ] リトライロジックはあるか？

### 完了報告

- [ ] デプロイ URL の取得方法は？
- [ ] コミット情報・CI 所要時間は報告に含まれるか？
- [ ] 失敗時のエスカレーション先は明記されているか？

## コード生成・変換系

### 入出力

- [ ] 入力は何か？（ユーザー指定、テンプレート、既存コード）
- [ ] 出力先のファイルパスは？
- [ ] 既存ファイルの上書き確認はあるか？

### 品質

- [ ] 生成コードはプロジェクトのコーディング規約に準拠しているか？
- [ ] このプロジェクト固有のルールは反映されているか？
  - CSS: oklch() のみ
  - Svelte 5: Runes API のみ
  - i18n: ja.json / en.json 同時更新
- [ ] 型安全性は担保されているか？

### テンプレート

- [ ] テンプレートのカスタマイズポイントは明確か？
- [ ] プレースホルダーの置換ルールは定義されているか？
- [ ] テンプレートのバージョン管理はどうするか？

## 汎用系

### 目的

- [ ] このスキルは他のカテゴリに該当しないか再確認したか？
- [ ] スキルの境界（何をやって何をやらないか）は明確か？
- [ ] 将来的に特定カテゴリに移行する可能性はあるか？

### ユーザー体験

- [ ] スキルのトリガー方法は直感的か？
- [ ] 実行中の進捗表示は必要か？
- [ ] 対話的な入力は必要か？

## 共通チェック（全カテゴリ）

### 命名

- [ ] スキル名はケバブケース（hyphen-case）か？
- [ ] 名前から目的が推測できるか？
- [ ] 既存スキルと命名規則が統一されているか？

### 構造

- [ ] H1 タイトル + 説明文は存在するか？
- [ ] 手順は番号付きリストで記述されているか？
- [ ] コマンド例はコードブロック内に記載されているか？

### 整合性

- [ ] CLAUDE.md のスキルテーブルに追記されているか？
- [ ] 他のスキルとの依存関係は明記されているか？
  - 例: deploy-test は deploy の後に実行することを想定
