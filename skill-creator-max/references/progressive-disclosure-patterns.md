# 段階的開示パターン

スキルの情報量をユーザーの習熟度や必要性に応じて段階的に提示するパターン。

## 概念

**段階的開示（Progressive Disclosure）** とは、最初は要約・概要のみを提示し、
必要に応じて詳細を開示する情報設計パターン。

スキル定義では以下のメリットがある:
- Claude がスキルを解釈する際の負荷を最適化
- 単純なケースでは簡潔に、複雑なケースでは詳細に対応
- メンテナンス性の向上（詳細を独立セクションに分離）

## パターン 1: サマリー → 詳細

### 構造

```markdown
# skill-name

1行の概要説明。

## 手順

1. ステップA
2. ステップB
3. ステップC

## 詳細（必要時のみ参照）

### ステップA の詳細
...

### ステップB の詳細
...
```

### 実例: deploy.md

```markdown
# deploy

push → CI 完了 → Vercel デプロイ URL 取得を一連で実行。

## 前提チェック
(3 項目の簡潔なリスト)

## 手順
### 1. push (コマンド 1 行)
### 2. CI 完了待機 (コマンド + 分岐)
### 3. URL 取得 (コマンド + リトライ条件)
### 4. 結果レポート (テンプレート)

## 失敗時の対応 (テーブルで詳細パターン)
```

**段階:** 概要 → 手順 → サブステップ → 失敗パターン

## パターン 2: ハッピーパス → エラーパス

### 構造

```markdown
## 手順

1. <正常系の手順>
2. <正常系の手順>
3. <正常系の手順>

## 失敗時の対応

| 状況 | 対応 |
|------|------|
| パターンA | 対処A |
| パターンB | 対処B |
```

### メリット

- 正常系を先に理解できる
- エラー対応は問題発生時のみ参照
- Claude は正常系を優先的に実行し、失敗時にエラーパスを参照

## パターン 3: 概要テーブル → 個別詳細

### 構造

```markdown
## チェックポイント早見表

| 項目 | デスクトップ | モバイル |
|------|------------|---------|
| Grid | 4列 | 2列 |
| Bar | なし | あり |

## 詳細手順

### デスクトップテスト
(詳細な確認手順)

### モバイルテスト
(詳細な確認手順)
```

### 実例: ui-test.md

最後にチェックポイント早見表を置き、
本文の詳細手順で段階的に掘り下げる。

## パターン 4: 共通手順 → 環境固有手順

### 構造

```markdown
## 手順

### 共通
1. ステップ1（全環境共通）
2. ステップ2（全環境共通）

### ローカル環境
3a. ローカル固有のステップ

### 本番環境
3b. 本番固有のステップ
```

### 適用条件

- 同じスキルが複数の環境で動作する
- 環境差異が少なく、スキルを分割するほどではない

## パターン 5: デフォルト → カスタマイズ

### 構造

```markdown
## 手順

1. デフォルト設定で実行

## カスタマイズ（任意）

引数で以下をカスタマイズ可能:
- URL: デフォルト `https://www.cmd-gen.com`
- ビューポート: デフォルト 1280x900
```

### 実例: perf-test.md

```markdown
引数で URL が渡された場合はそちらを使用する。
```

引数なしでデフォルト動作し、必要に応じてカスタマイズ可能。

## パターン 6: マルチレベルヘッダー

### 構造

```markdown
## テスト手順（H2 = 大分類）

### 1. デスクトップテスト (1280x900)（H3 = 中分類）

- screenshot で以下を確認:（リスト = 詳細項目）
  - Header: ロゴ・タイトル
  - PresetGrid: 4列グリッド
```

### 深度ガイドライン

| レベル | 用途 | 例 |
|--------|------|------|
| H1 | スキル名 | `# ui-test` |
| H2 | 大セクション | `## テスト手順` |
| H3 | サブステップ | `### 1. デスクトップテスト` |
| リスト | 詳細項目 | `- Header: ロゴ・タイトル` |
| サブリスト | 最詳細 | `  - テーマトグルのアイコン` |

**推奨**: H3 までに抑える。H4 以降は使わず、リストで表現する。

## アンチパターン

### NG: フラット構造（段階なし）

```markdown
## 手順

1. npm run check で型チェック。エラーがあれば修正する。よくあるエラーは
   TS2345（型不一致）でこれは引数の型を確認して...。CI でも同じチェック
   が走るので...。（長い1段落に全情報を詰め込み）
```

**問題**: 正常系とエラー対応が混在し、読みにくい

### NG: 過度な分割

```markdown
## 概要
## 前提
## 環境設定
## 手順 Phase 1
## 手順 Phase 2
## 手順 Phase 3
## エラー Phase 1
## エラー Phase 2
## エラー Phase 3
## レポート形式
## 例外ケース
## FAQ
```

**問題**: セクションが多すぎて全体像を把握しにくい

### 推奨: バランスの良い構造

```markdown
## 前提
## 手順
### 1. xxx
### 2. xxx
### 3. xxx
## 失敗時の対応
## 結果レポート
```

**H2: 3-5 セクション、H3: 各 H2 に 2-5 サブセクション** が目安。
